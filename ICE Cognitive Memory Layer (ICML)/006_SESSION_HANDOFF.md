\# ICE™ SESSION HANDOFF



\*\*Guía Oficial para Continuar el Proyecto en una Nueva Sesión\*\*



\*\*Versión:\*\* 1.0

\*\*Estado:\*\* Activo



\---



\# Objetivo



Este documento permite reanudar el desarrollo de ICE™ sin pérdida de contexto.



Antes de comenzar cualquier nueva sesión, una IA o colaborador deberá leer este documento y los archivos contenidos en la carpeta `/memory`.



Su finalidad es evitar repetir discusiones, replantear decisiones ya aprobadas o perder tiempo reconstruyendo el estado del proyecto.



\---



\# Estado Actual del Proyecto



ICE™ ha completado su fase de definición estratégica.



Actualmente el proyecto dispone de:



\* Visión y misión consolidadas.

\* North Star definida.

\* Business Blueprint aprobado.

\* PRD aprobado.

\* Arquitectura general definida.

\* Memoria persistente del proyecto.

\* Repositorio GitHub ICE\_OS operativo.

\* Landing pública disponible.

\* Primer prototipo funcional del ICE Beacon.



La siguiente etapa corresponde al desarrollo progresivo del MVP.



\---



\# Documentos que Deben Revisarse



Antes de iniciar cualquier trabajo deberán leerse, en este orden:



1\. `000\_PROJECT\_CONTEXT.md`

2\. `001\_DECISIONS.md`

3\. `002\_ARCHITECTURE\_STATE.md`

4\. `003\_PRODUCT\_STATE.md`

5\. `004\_NEXT\_SESSION.md`

6\. `005\_LESSONS\_LEARNED.md`



Estos documentos representan la memoria oficial del proyecto.



\---



\# Principios de Trabajo



Toda nueva sesión deberá respetar las siguientes reglas:



\* No replantear decisiones ya aprobadas.

\* No reiniciar análisis estratégicos salvo solicitud expresa del fundador.

\* Construir sobre el trabajo existente.

\* Mantener la documentación sincronizada con el estado real del proyecto.

\* Priorizar entregables verificables sobre discusiones extensas.



\---



\# Metodología de Desarrollo



Cada sesión seguirá el siguiente ciclo:



1\. Revisar la memoria del proyecto.

2\. Verificar el estado del repositorio GitHub.

3\. Definir un único objetivo.

4\. Construir el entregable.

5\. Validar el resultado.

6\. Actualizar la documentación correspondiente.

7\. Preparar el siguiente punto de continuidad.



\---



\# Fuente Oficial de Verdad



La información oficial del proyecto reside en el repositorio \*\*ICE\_OS\*\*.



Las conversaciones sirven para desarrollar nuevas ideas y producir entregables, pero la memoria permanente del proyecto se conserva en GitHub.



Toda modificación relevante deberá quedar documentada antes de cerrar una sesión.

\# Forma de Evaluar el Progreso



El avance del proyecto se medirá mediante resultados concretos.



Ejemplos:



\* componentes implementados;

\* código funcional;

\* documentos aprobados;

\* versiones publicadas;

\* commits registrados.



El número de conversaciones mantenidas no constituye un indicador de avance.

2\. Construir el Executive Assessment.

5\. Evolucionar la landing hacia una experiencia interactiva.

\---



\# Reglas para Futuras Decisiones



Toda nueva decisión deberá:



\* aportar valor al empresario;

\* respetar la arquitectura vigente;

\* alinearse con la visión del proyecto;

\* quedar registrada en `001\_DECISIONS.md`.



Si una decisión modifica el estado del producto o de la arquitectura, deberán actualizarse también los documentos correspondientes.



\---



\# Criterios de Calidad



Antes de considerar finalizada una sesión deberán cumplirse las siguientes condiciones:



\* existe al menos un entregable verificable;

\* la documentación refleja el nuevo estado del proyecto;

\* GitHub puede actualizarse sin pérdida de información;

\* el siguiente paso del proyecto está claramente definido.



\---



\# Mensaje para la Próxima IA



ICE™ no es únicamente un proyecto de software.



Es la materialización del conocimiento acumulado durante décadas de experiencia en consultoría, implantación de ERP y dirección tecnológica.



Cada nueva contribución deberá preservar ese conocimiento y transformarlo en un producto útil, confiable y orientado al empresario.



El objetivo no es desarrollar más funcionalidades.



El objetivo es ayudar a empresarios a tomar mejores decisiones.

A partir de este momento:



\* GitHub será la memoria oficial.

\* Los documentos de `/memory` deberán mantenerse sincronizados.

\* Cada sesión actualizará únicamente aquello que haya cambiado.

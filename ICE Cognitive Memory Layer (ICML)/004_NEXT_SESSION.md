# ICE™ NEXT SESSION

**Plan Oficial de Continuidad del Proyecto**

**Versión:** 1.0
**Estado:** Activo

---

# Objetivo del Documento

Este documento tiene como finalidad permitir que cualquier nueva sesión de trabajo continúe exactamente donde terminó la anterior.

Debe actualizarse al finalizar cada sesión importante y convertirse en la guía de trabajo inmediata para el siguiente encuentro.

---

# Estado Actual del Proyecto

El proyecto ha completado exitosamente su fase de definición estratégica.

Actualmente se dispone de:

* Visión del proyecto.
* Misión.
* North Star.
* Business Blueprint.
* PRD.
* Arquitectura general.
* Memoria persistente del proyecto.
* Repositorio GitHub (ICE_OS).
* Landing pública operativa.
* Primer prototipo del componente ICE Beacon.

La siguiente etapa corresponde principalmente al desarrollo del producto.

---

# Objetivo Principal de la Próxima Sesión

**Transformar el ICE Beacon en una experiencia conversacional de alto valor para el empresario.**

El objetivo no es mejorar únicamente el aspecto visual del componente.

El propósito es convertirlo en el primer contacto inteligente entre el visitante e ICE™.

---

# Prioridad Nº 1

## Completar el ICE Beacon

El componente deberá:

* integrarse completamente con la landing existente;
* respetar la identidad visual del sitio;
* funcionar correctamente en dispositivos móviles;
* permitir iniciar una conversación breve;
* despertar curiosidad sin generar falsas expectativas.

---

# Flujo Esperado

El visitante deberá experimentar el siguiente recorrido:

Landing

↓

ICE Beacon

↓

Pregunta Ejecutiva

↓

Respuesta

↓

Reflexión personalizada

↓

Invitación al Executive Assessment

Este flujo constituye la primera experiencia oficial de ICE™.

---

# Criterios de Aceptación

El ICE Beacon se considerará terminado cuando:

* aparezca correctamente;
* pueda cerrarse;
* pueda abrirse nuevamente;
* funcione en móviles;
* presente una pregunta ejecutiva;
* adapte el mensaje según la respuesta del visitante;
* no prometa funcionalidades inexistentes.

---

# Prioridad Nº 2

## Construcción del Executive Assessment

Una vez completado el ICE Beacon, la siguiente prioridad será diseñar el primer Executive Assessment.

Este deberá incluir:

* preguntas claras;
* duración inferior a cinco minutos;
* lenguaje ejecutivo;
* orientación práctica.

El resultado será la base para el Executive Score™.

---

# Prioridad Nº 3

## Executive Score™

Definir el modelo de puntuación que utilizará ICE™.

Aspectos a desarrollar:

* escala de evaluación;
* categorías;
* interpretación;
* representación visual.

---

# Prioridad Nº 4

## Executive Report™

Diseñar el primer informe ejecutivo generado automáticamente.

Debe incluir:

* resumen general;
* fortalezas;
* oportunidades;
* riesgos;
* recomendaciones.

---

# Entregables Esperados

Las próximas sesiones deberán generar activos verificables.

Ejemplos:

* código HTML;
* archivos CSS;
* componentes JavaScript;
* documentos Markdown;
* commits;
* nuevas versiones.

Las sesiones no deberán finalizar únicamente con ideas o propuestas.

---

# Regla de Trabajo

Antes de iniciar una nueva conversación deberá verificarse:

* que todos los documentos de la carpeta `/memory` estén actualizados;
* que GitHub refleje el estado real del proyecto;
* que exista un objetivo único para la sesión.

No deberán iniciarse múltiples líneas de trabajo simultáneamente.

---

# Metodología

Cada sesión seguirá el siguiente ciclo:

1. Revisar el estado actual.
2. Definir un único objetivo.
3. Construir el entregable.
4. Validar el resultado.
5. Actualizar la memoria del proyecto.
6. Realizar commit en GitHub.

---

# Riesgos Identificados

Durante las primeras fases del proyecto se identificaron los siguientes riesgos:

* exceso de planificación (overplanning);
* exceso de análisis (overthinking);
* pérdida de contexto entre sesiones;
* dependencia de la memoria del chat.

Las acciones implementadas para mitigarlos son:

* documentación persistente;
* uso de GitHub como fuente de verdad;
* objetivos concretos por sesión;
* generación de entregables verificables.

---

# Indicador de Éxito

Una sesión será considerada exitosa únicamente si produce al menos uno de los siguientes resultados:

* código funcional;
* componente terminado;
* documento aprobado;
* commit listo para GitHub;
* mejora visible del producto.
